# Table-Stakes Checklist — Third-Party iOS Keyboard Requirements

*Researched 2026-07-15. Scope: features/requirements a third-party iOS keyboard MUST or SHOULD
have, that PLAN.md doesn't already cover, or covers only partially. Judged against PLAN.md v1
scope + current repo state (App/, KeyboardExt/, entitlements, Packages/KeyboardKit vendored
9.9.1). Does not re-litigate anything already decided in PLAN.md or re-research SwiftKey pain
points (see research/swiftkey-frustrations.md).*

Status legend: **covered** (in PLAN.md or implemented), **partial** (some plan/code exists but
gap remains), **missing** (not mentioned anywhere).

## 1. App Store compliance

| Feature / requirement | Why table-stakes | Status | Milestone |
|---|---|---|---|
| Keyboard functions fully with Full Access **denied** | Guideline 4.4.1: keyboard extensions must "remain functional without full network access and without requiring full access." Apple has rejected keyboards that silently degrade to nonfunctional without Full Access. Our architecture (extension never networks; App Group only needed for learning/sync) already satisfies this by design. | **covered** (PLAN.md: "Extension untouched by Full Access denial → degrades to local-only learning, still fully functional") | v1-blocker (verify explicitly before submission — add a manual QA pass: deny Full Access, confirm typing/autocorrect/prediction still work, only sync/learning-persistence degrades) |
| `RequestsOpenAccess=YES` justified in review notes | Setting this flag is a known reviewer trigger — Apple explicitly asks *why* a keyboard needs it and checks the behavior actually matches (App Group sharing, in our case, not networking). Undocumented/unjustified use has caused rejections historically. | **partial** — flag is already set in `KeyboardExt/Info.plist`; no App Store Connect review-notes text exists yet since we haven't submitted | v1-blocker |
| Privacy Policy URL (required for any app requesting Full Access / on the "Sensitive Data" path) | App Store Connect requires a privacy policy URL for apps handling user data of any kind, and Full Access keyboards are scrutinized closely regardless of actual data use. | **missing** — no privacy policy drafted or hosted; App/Info.plist has no policy URL | v1-blocker |
| App Privacy "nutrition label" declarations — declare **"Data Not Collected"** | Apple's privacy label (App Store Connect → App Privacy) is a self-declared questionnaire; a true zero-telemetry keyboard should show the rare/trust-building "Data Not Collected" label. Declaring wrong (e.g., missing that CloudKit sync exists, even if it's the *user's own* iCloud, not ours) risks removal for privacy-label inaccuracy — Apple has enforced this before. Need to get the declaration technically correct: user content synced to the user's own CloudKit private DB, initiated by the user, not "linked to identity" in Apple's taxonomy, generally answerable as not collected by *us*, but the questionnaire has a nuance around "data used to provide sync/backup" that should be filled with care. | **missing** — no App Store Connect app record exists yet, so no declaration made | v1-blocker |
| Export compliance declaration for CryptoKit AES-GCM sync payloads | `ITSAppUsesNonExemptEncryption` in Info.plist. Apple's own CryptoKit APIs used per Apple's intended purpose (not proprietary/altered crypto) generally qualify for the standard **exemption** (no BIS filing needed) — but the key still needs declaring, or App Store Connect blocks submission/TestFlight distribution with a "missing compliance" warning every build. | **missing** — no `ITSAppUsesNonExemptEncryption` key in App/Info.plist yet (sync isn't built, M3); flag it when CryptoKit lands | v1-blocker (blocks any TestFlight build once sync code lands, not just final submission) |
| App Tracking Transparency (ATT) prompt — **not required** | ATT only triggers for IDFA/cross-app tracking. A keyboard that does zero analytics, zero ad SDKs, zero cross-app identifiers has nothing to declare and should never show the ATT prompt. Worth stating explicitly in-app/in-review-notes so reviewers don't wonder why it's absent (a keyboard silently *not* asking for tracking is unusual enough among incumbents that it's worth calling out as a feature, not an oversight). | **covered** (by having zero telemetry — PLAN.md decision "zero network code" already implies no ATT surface) | v1-nice (explicit no-ATT statement in privacy policy / About screen, for reviewer clarity and user trust) |
| Guideline 4.4.1 sub-requirements: keyboard must provide a method to switch to next keyboard (globe key), and if it displays images/emoji it must follow Guideline 1.1.4/Sticker content rules | Baseline functional requirements, easy to miss in a fresh build. | **covered** — globe key is in v1 bottom-row scope; no emoji keyboard shipped (KeyboardKit's stock emoji picker is present in the view hierarchy per `KeyboardViewController.swift`'s `emojiKeyboard: { $0.view }`, so this rule does technically apply — the emoji content itself is Apple's system set, not custom, which is the normal case and not a rejection risk) | v1-blocker to confirm globe key always present in every keyboard-type row (not just alphabetic) |

**Sources:** [Apple App Extension Programming Guide: Custom Keyboard](https://developer.apple.com/library/archive/documentation/General/Conceptual/ExtensibilityPG/CustomKeyboard.html), [Configuring open access for a custom keyboard](https://developer.apple.com/documentation/uikit/configuring-open-access-for-a-custom-keyboard), [ITSAppUsesNonExemptEncryption docs](https://developer.apple.com/documentation/bundleresources/information-property-list/itsappusesnonexemptencryption), [App Store Connect export compliance guides](https://orbitkit.io/blog/app-store-export-compliance-encryption/), Apple App Review Guidelines (developer.apple.com/app-store/review/guidelines).

## 2. Onboarding / app UI table stakes

| Feature / requirement | Why table-stakes | Status | Milestone |
|---|---|---|---|
| Deep-link to Settings for enabling the keyboard | Users expect a one-tap path, not "go find Settings yourself." The only **documented, supported** mechanism is `UIApplication.openSettingsURLString` (opens the app's own Settings page, one level short of the actual Keyboards list — user still taps through General → Keyboard → Keyboards themselves). Undocumented deep paths (`App-Prefs:`/`prefs:root=General&path=Keyboard`) are explicitly against guidelines except for the one Apple-sanctioned case (opening keyboard settings *from inside the running extension itself*, via `UIInputViewController.advanceToNextInputMode`-adjacent `openURL` trick documented in Apple QA1924) — not usable from the *containing app*, and even that is deprecated/unstable across iOS versions. Current state of the art (2026) has not improved past `openSettingsURLString`. | **covered** — `ContentView.swift` already opens Settings via `openSettingsURLString`, which is the best available option; do not add undocumented `prefs:`-scheme paths | v1-blocker to keep as-is; note in code that this is the ceiling of what's possible, not a shortcut worth "fixing" |
| Keyboard-enabled detection from the containing app | Would allow showing "already set up!" instead of always showing onboarding steps. **No official API exists.** The one documented community trick — reading the global `UserDefaults.standard` `"AppleKeyboards"` array of bundle IDs — is undocumented, Apple-forums-sourced, and has a history of silently breaking across iOS versions (sandboxing tightened more than once); relying on it risks either false negatives (showing setup steps forever to a configured user) or worse, false confidence. | **missing** — PLAN.md doesn't mention this at all; current onboarding always shows the 3-step flow regardless of actual state | v1-nice — if implemented, must have a graceful fallback (assume "not yet enabled" and always show the steps + button) since the API is unofficial; never gate core functionality on it |
| Full Access explainer screen (honest, works-without-it framing) | Users are primed to distrust "Allow Full Access" (it's the single most privacy-anxious moment in keyboard setup, per the SwiftKey/Fleksy comparison — Fleksy markets "no Full Access needed for predictions" as a selling point; reputable actors explain *exactly* what Full Access is for and confirm the app works without it). PLAN.md defers this to M3 ("onboarding, Full Access explainer") but gives no content/UX spec yet. | **partial** — milestone exists (M3), no explainer copy or screen designed yet | v1-blocker (for the M3 milestone specifically — should ship before public TestFlight, since this is the exact trust moment competitors get wrong) — concrete content needed: state plainly that Full Access enables (a) reading/writing the App Group container for personal-dictionary sync, (b) nothing else (no network code in the binary — auditable), and that autocorrect/prediction/typing work identically without it, only sync/cross-device learning is disabled |
| Test-typing field in onboarding | Lets users confirm the keyboard actually works before leaving the app; standard pattern. | **covered** — `ContentView.swift` has a `TextField` with `sampleText`/focus | done |
| "Everything already configured" state | When keyboard is enabled + Full Access granted, onboarding should not nag; should show a settings-style summary instead (or hide the setup steps). Ties to the "keyboard-enabled detection" gap above — without reliable detection this can't be built cleanly. | **missing** | v1-nice, blocked on (and lower priority than) the detection question above; acceptable v1 fallback: always show steps but make them visually "check-offable" by the user manually, no auto-detection required |
| In-app Full Access *current status* indicator (granted/not) | This one IS detectable and documented: `UIPasteboard.general` / more commonly checking whether the extension can read the shared App Group `UserDefaults`/file container is exactly what Full Access gates, so the **app side** can infer status indirectly by checking whether the extension has written anything to the App Group recently, or the containing app can call a documented check — actually the standard trick here is the extension writing a heartbeat/flag to the App Group once per invocation, which the app reads. This is different from "is the keyboard enabled" (no clean API) — Full Access status specifically can be inferred this way with reasonable reliability. | **missing** | v1-nice, useful supplementary signal alongside the Full Access explainer (M3) |

**Sources:** [Apple QA1924: Opening Keyboard Settings from a Keyboard Extension](https://developer.apple.com/library/archive/qa/qa1924/_index.html), [Configuring open access for a custom keyboard](https://developer.apple.com/documentation/uikit/configuring-open-access-for-a-custom-keyboard), Stack Overflow / community `AppleKeyboards` UserDefaults technique (undocumented, cross-referenced across multiple threads, consistently caveated as unofficial), [Securing.pl: Third-party iPhone keyboards vs security](https://www.securing.pl/en/third-party-iphone-keyboards-vs-your-ios-application-security/) (SwiftKey/Fleksy Full-Access design comparison).

## 3. Dictionary / data lifecycle

| Feature / requirement | Why table-stakes | Status | Milestone |
|---|---|---|---|
| Export my data | Users of a privacy-first product expect data portability as a first-class right, not an afterthought — this is the flip side of "your data, your iCloud." SwiftKey's own export is a flat JSON: `{"last_sync": ..., "terms": ["word1", "word2", ...]}` — a reasonable convention to mirror (simple array of learned + user-added words; could extend with per-word frequency counts given our richer learning model). | **missing** — DictionaryView.swift has *import* (SwiftKey vocabulary.txt) but no export | v1-blocker — asymmetric "we'll take your SwiftKey export but never give you one back" undermines the entire trust pitch |
| Reset / clear ALL learned data (one destructive action) | SwiftKey precedent: Account → Data Settings → "Remove my remote data" / "Delete backup" — a full wipe is the expected escape hatch, and Microsoft's own support explicitly offers it as the *only* remedy for a corrupted/unwanted learned model since individual removal has historically been incomplete on their iOS build. Our per-word delete already beats that SwiftKey pain point (PLAN.md #8), but the emergency "start over" nuclear option is still expected UI even when per-word deletion exists — it's the fast path after e.g. handing the phone to a kid, or a corrupted personal model. | **missing** — DictionaryView has per-word delete + undo, no "delete everything" action | v1-blocker |
| iCloud sync opt-out | PLAN.md's sync is "zero-config, no account" (CloudKit private DB) — good default, but users must be able to turn it off entirely (pure local-only mode) for those who don't trust iCloud or share an Apple ID. Table stakes for any "your data, your call" pitch — otherwise the privacy story has a silent mandatory-cloud asterisk. | **missing** — PLAN.md's sync design has no opt-out mentioned; SettingsView.swift's sync section is just a "coming soon" placeholder with no toggle | v1-blocker (tie to M3, the sync milestone — build the toggle in the same wave as sync itself, not after) |
| What happens on app uninstall | Users expect (and competitors get wrong) clarity on this. Concretely for our architecture: extension + app are one bundle — uninstalling deletes *both* together, and the shared App Group container is cleaned up once no member of the group remains installed (unlike separate-app + widget scenarios where the container can outlive one member). So local learned data + local App Group state is fully gone on uninstall. **But** anything already synced to CloudKit survives independently in the user's iCloud (by design — that's the backup/multi-device value) — meaning reinstalling restores from iCloud, and a user who *wants* a truly clean slate needs the explicit "delete everything" action (previous row) to also nuke the CloudKit records, not just rely on uninstalling. This asymmetry needs explicit, honest copy somewhere (About/Settings/Privacy Policy) — "uninstalling deletes local data; your synced dictionary remains in your iCloud until you delete it here" — otherwise it's a surprise, not a feature. | **missing** — not addressed in PLAN.md or in-app copy | v1-blocker (documentation/copy, not code — pairs naturally with the "reset all data" action, which should be the thing that also clears CloudKit records) |
| Blocklist / offensive-word handling in base lexicon | Users expect *some* baseline restraint from autocorrect/prediction proactively suggesting profanity mid-typo-correction (Gboard's "Block Offensive Words" toggle is the concrete precedent — on by default, user-togglable), while never *blocking* a correctly-typed word the user actually wants (SwiftKey doesn't replace "fucking" with "ducking" — it just avoids *suggesting* profanity as an autocorrection of a typo). For IS+EN this needs its own base-lexicon curation pass (BÍN/lemma-is likely already excludes or under-weights slurs by virtue of corpus frequency, but this hasn't been verified) — separate from the *learned* dictionary, where PLAN.md's user-editable/deletable design already handles "never suggest this word again" for anything the user personally blocklists. | **missing** for the base-lexicon dimension (no stated policy on profanity in bundled BÍN/EN wordlists, no toggle); **partial** for user-side (per-word delete already lets a user suppress an unwanted *learned* suggestion, but there's no explicit "never suggest even if it re-enters via base lexicon" primitive — deleting a learned word doesn't block the base-model form from resurfacing) | v1-nice — audit bundled lexicons for slur/profanity handling before v1 ship (verify BÍN/lemma-is doesn't autocorrect innocuous typos into slurs); a togglable "block offensive words" default-on setting is reasonable post-v1 polish, not a blocker, since the risk (autocorrecting *into* profanity) is lower-severity than SwiftKey/Gboard's original problem (voice dictation asterisk-censoring, which we don't have since no dictation) |
| iOS system text replacements (Settings → General → Keyboard → Text Replacement) | Users with existing shortcuts (e.g. "omw" → "on my way") expect **any** keyboard, including third-party ones, to expand them — this is a well-known SwiftKey/Gboard gap users complain about. Apple's supported mechanism is `UILexicon` via `requestSupplementaryLexicon(completion:)`, available in keyboard extensions **without Full Access** — it returns contact names *and* the user's Text Replacement shortcuts (`UILexiconEntry.documentText`/`userInput` pairs), explicitly documented as one of its two data sources. This is a real, currently-unused capability gap. | **missing** — no `UILexicon` integration mentioned in PLAN.md or present in KeyboardExt code | v1-nice — no Full Access required, moderate engineering lift (fetch once per session, merge into TypeEngine's candidate set as exact-match high-priority substitutions), meaningfully closes a documented incumbent gap |

**Sources:** [Microsoft SwiftKey Data Portal support](https://support.microsoft.com/en-us/swiftkey-keyboard/microsoft-swiftkey-keyboard-data-portal), [SwiftKey "remove my data" support article](https://support.swiftkey.com/hc/en-us/articles/201468321-How-can-I-remove-my-data-and-the-consent-to-use-my-data-), [App Group data persistence explainer](https://www.codestudy.net/blog/uninstalling-app-not-delete-app-group-data-do-i-have-to-remove-app-group-container-and-it-s-content-manually/), [Gboard "Block Offensive Words" toggle](https://support.google.com/android/thread/239494617/unblock-offensive-words-on-gboard), UILexicon docs (developer.apple.com), research/swiftkey-frustrations.md pain #8 (already in repo).

## 4. Keyboard-surface table stakes

| Feature / requirement | Why table-stakes | Status | Milestone |
|---|---|---|---|
| Caps lock (double-tap shift) | Universal iOS keyboard muscle memory. | **covered** — confirmed in vendored `Packages/KeyboardKit/Sources/KeyboardKit/_Keyboard/Keyboard+StandardKeyboardBehavior.swift` (`isDoubleShiftTap` / `doubleTapThreshold` logic feeding `.capsLocked` in `KeyboardContext+KeyboardCase.swift`) — free, unmodified KeyboardKit behavior, no code needed from us | done |
| Autocapitalization modes (sentence/word/none, per field type) | Standard iOS expectation, especially correct behavior in URL/email fields (already a PLAN.md concern for autocorrect, but autocap specifically isn't called out). | **partial** — KeyboardKit's `KeyboardContext` machinery (`KeyboardContext+Sync.swift`, `KeyboardSettings.swift`) provides this out of the box (reads `UITextAutocapitalizationType` from the host field), unmodified by our code — functionally covered by the framework, but PLAN.md doesn't explicitly claim it as tested/verified, worth a QA checklist line item | v1-blocker to verify (not to build) |
| Dark/light appearance following host app (`UIKeyboardAppearance`) | Table stakes — a keyboard that stays light-themed in a dark-mode host app looks broken. | **covered** — confirmed in `KeyboardContext.swift`: `colorScheme` synced from `traitScheme`, and `keyboardAppearance` reads `textDocumentProxy.keyboardAppearance ?? .default` directly — free KeyboardKit behavior | done |
| Long-press symbols on letter keys | PLAN.md already claims this as a differentiator vs SwiftKey iOS ("SwiftKey iOS never shipped this") — but currently only accented vowels (á é í ó ú ý) + ð/þ are wired via `Callouts.Actions.icelandic`; no *symbol* long-press (e.g. numbers on letter row, common on Android Gboard/SwiftKey) is implemented. Worth clarifying PLAN.md's claim refers specifically to **accent** long-press, not full symbol-on-letter long-press (a different, larger feature: e.g. long-press `e` → `3`, `r` → `4`, etc., Android muscle memory some iOS switchers expect but iOS-native users don't). | **partial** — accent long-press done; number/symbol-on-letter long-press not planned or built | v1-nice (low priority — this is an Android-muscle-memory expectation more than an iOS one; iOS users are used to a dedicated 123 key, which we have) |
| "." key + long-press cluster | Already in v1 scope. | **covered** | done |
| Number row (persistent, Gboard-style toggle) | Some power users (esp. those coming from Android/Gboard) expect an optional always-visible number row above the letters. Not standard iOS behavior (stock keyboard doesn't have it either), so it's a nice-to-have differentiator, not baseline expectation for an *iOS* keyboard. No number-row code found anywhere in vendored KeyboardKit 9.9.1 (`grep` for `numberRow`/`NumberRow` returns nothing). | **missing**, and would require net-new layout work (not a KeyboardKit freebie) | post-v1 |
| One-handed / keyboard-width mode | Apple's stock keyboard has had one-handed mode since iOS 11; some users expect parity. No one-handed support found anywhere in vendored KeyboardKit 9.9.1 source (`grep` for `oneHanded`/`OneHand` across `Sources/` returns nothing) — likely a v10/Pro-era or not-yet-built feature, not something free-tier 9.9.1 gives us. | **missing** | post-v1 (nontrivial — would need custom layout-width logic; low usage relative to build cost, and Apple's own system keyboard remains available as a fallback for one-handed use since the user can switch keyboards) |
| Haptic feedback — **works without Full Access** claim needs re-verification | PLAN.md v1 scope lists "haptics + key sounds" as a feature but doesn't address the Full Access dependency. Multiple independent sources (Apple Developer Forums, WWDC-adjacent community writeups) confirm `UIImpactFeedbackGenerator`/`UIFeedbackGenerator` **do not produce haptic output inside a keyboard extension unless Full Access is granted** — this is a documented, longstanding iOS sandboxing limitation (extensions without Full Access are blocked from the haptic engine), not something fixable app-side. This directly contradicts the "extension untouched by Full Access denial → fully functional" claim in PLAN.md's Sync section *if* haptics are read as part of "fully functional" — they will silently do nothing without Full Access. | **partial / needs correction** — PLAN.md's haptics feature has no caveat about this; risks user confusion ("haptics don't work") when Full Access is denied, unless messaged | v1-blocker — must (a) confirm this limitation still holds on current iOS via a device test with Full Access off, and (b) if confirmed, add explicit copy to the Full Access explainer: "haptic/typing-sound feedback requires Full Access; typing, autocorrect, and prediction do not" — key sounds (`AudioServicesPlaySystemSound`) are a separate API and *may* not share this restriction — verify independently, don't assume sound and haptics behave identically |
| Landscape layout | Should render sanely (KeyboardKit's device-configuration layer, `KeyboardLayout+DeviceConfiguration.swift`, exists specifically for this — screen-width-based row sizing). No custom landscape work has been done for the Icelandic layout specifically. | **partial** — inherited for free from KeyboardKit's layout engine, no Icelandic-specific landscape testing done | v1-blocker to verify (not build) — add to device-QA checklist |
| iPad — "functional but unoptimized" | Explicit PLAN.md decision #3. No further action needed beyond what's already decided, just noting it's consistent with this checklist's other iPad-adjacent findings (one-handed mode, number row are also lower iPad priority). | **covered** (by decision) | n/a |
| Emoji access via globe-to-system-emoji | PLAN.md explicitly excludes an emoji keyboard/search (non-goal). The vendored KeyboardKit `KeyboardView` still renders its own stock `emojiKeyboard: { $0.view }` slot (confirmed in `KeyboardViewController.swift`) — meaning we *do* ship KeyboardKit's built-in emoji picker today, contradicting the "no emoji keyboard" framing unless that's intentional. Users overwhelmingly expect *some* emoji access from any modern keyboard; globe-key-to-system-emoji-keyboard (Apple's own picker) is the honest, zero-maintenance path if we deliberately strip KeyboardKit's own emoji view. Also flagged in PLAN.md's own Risks section: "Emoji keyboard font-cache memory bloat (KK GH #757, unresolved) — watch under jetsam budget." | **partial / needs a decision** — PLAN.md's non-goal ("emoji search") and the actual shipped code (KeyboardKit's stock emoji view, memory-risky per GH #757) are in tension; either (a) deliberately keep KeyboardKit's basic emoji grid (accept the GH #757 memory risk, gains "some emoji access" for free) or (b) rip it out and rely purely on the globe key routing to Apple's system emoji keyboard (zero emoji-related memory risk, matches "no emoji" non-goal cleanly) | v1-blocker to *decide* explicitly — currently ambiguous by omission, and the memory risk in a jetsam-capped extension is exactly the kind of thing that should be a conscious choice, not a default inherited from the framework |
| Dictation absence (no mic key) — honest messaging | PLAN.md non-goals list "voice dictation" as excluded — correct call given the mic key requires Full Access + Apple's own Speech framework/dictation entitlement in an extension context (nontrivial, arguably impossible cleanly without Full Access + extra entitlements) — but no user-facing messaging plan exists for *why* the mic key is absent, when every incumbent (Apple, SwiftKey, Gboard) has one. Users will notice the missing mic key and may assume it's broken vs. intentional. | **missing** (messaging, not the feature) | v1-nice — one line in the About/Settings screen ("no voice dictation — keeps the extension network-free and Full-Access-optional for typing") turns an absence into a stated trade-off instead of an apparent bug |
| Password / secure-field behavior | iOS **automatically and unconditionally** swaps to the system keyboard whenever `secureTextEntry` is true, then swaps back — this is platform-enforced, not something any third-party keyboard can opt into or fix. Users experience this as "the keyboard keeps disappearing" if not told it's expected. Directly named in swiftkey-frustrations.md (#11) as a top platform-limitation complaint that compounds frustration even though it isn't the app's fault. | **missing** messaging — PLAN.md doesn't mention this at all, no in-app copy planned | v1-blocker (copy only, no code possible) — add a line to onboarding/About: "You'll briefly see Apple's keyboard in password fields — this is an iOS security rule for every third-party keyboard, not a bug" |
| VoiceOver / accessibility on the custom keyboard surface | App Review's own VoiceOver evaluation criteria (App Store Connect → Manage App Accessibility) and general HIG guidance expect custom controls (which every key on a custom keyboard is) to have correct accessibility labels/traits and be operable via VoiceOver gestures. KeyboardKit 9.9.1 ships `KeyboardAction+Accessibility.swift` (confirmed present in vendored source) providing baseline accessibility labels for standard actions — a real, unearned-by-us baseline — but the Icelandic-specific characters (ð, þ, æ, ö) and our custom `.` key/callout cluster have not been verified to have correct/pronounceable VoiceOver labels (e.g., does VoiceOver read "ð" understandably, or as "d with stroke" vs. the Icelandic name "eð"?). | **partial** — baseline inherited free from KeyboardKit; Icelandic-specific labels unverified | v1-blocker to verify (VoiceOver pass over the Icelandic-specific keys specifically), not a build item unless labels are found to be wrong |

**Sources:** [Apple Developer Forums: haptic feedback in keyboard extensions](https://developer.apple.com/forums/thread/63493), [UIImpactFeedbackGenerator docs](https://developer.apple.com/documentation/uikit/uiimpactfeedbackgenerator), [Configuring a custom keyboard interface — secure text entry behavior](https://developer.apple.com/documentation/uikit/configuring-a-custom-keyboard-interface), [VoiceOver evaluation criteria — App Store Connect](https://developer.apple.com/help/app-store-connect/manage-app-accessibility/voiceover-evaluation-criteria/), local inspection of `Packages/KeyboardKit/Sources/KeyboardKit/_Keyboard/` (KeyboardContext.swift, Keyboard+StandardKeyboardBehavior.swift, KeyboardContext+KeyboardCase.swift, Actions/KeyboardAction+Accessibility.swift) and `KeyboardExt/KeyboardViewController.swift`, PLAN.md Risks section (GH #757 emoji memory issue, already flagged there but not resolved into a decision).

## 5. Trust surface

| Feature / requirement | Why table-stakes | Status | Milestone |
|---|---|---|---|
| Link the open-source repo in-app | The entire pitch is "audit the repo, it can't phone home" (PLAN.md self-grill section) — that claim is only actionable if users can actually find and open the repo. SettingsView.swift's "Um Lyklaborð" (About) section currently states open-source/BÍN-attribution/no-telemetry copy but (per the code read) has no tappable link to the actual GitHub URL. | **partial** — the *claim* is made in copy, the *link* is missing | v1-blocker (trivial to add, disproportionately high trust value — this is the single cheapest, highest-leverage trust action available) |
| "Network access: none" statement referencing iOS's own enforcement | Rather than asking users to simply trust a claim, point out that iOS's own extension sandbox *enforces* no-network by default (Full Access is what would be needed to lift it, and even then no networking code exists in the extension) — i.e., frame it as a verifiable platform fact, not a promise. This is a stronger trust claim than most competitors make, since it's independently checkable (someone could theoretically pcap the device and confirm). | **missing** as explicit copy — the underlying architectural fact is true and repeatedly stated in PLAN.md, but not yet surfaced as user-facing copy that references iOS's *enforcement* specifically (vs. just "we don't track you") | v1-nice — strengthens the existing "aboutNoTelemetryDetail" copy already in SettingsView.swift with the platform-enforcement angle, not a new feature |
| Reproducible builds | The strongest possible trust mechanism (verify the App Store binary matches the published source) — but genuinely hard on iOS: Apple FairPlay-encrypts every App Store binary at upload, and Apple's closed toolchain (Xcode/ld/Swift compiler determinism) makes bit-for-bit reproducibility much harder than on Android (where F-Droid's build server model works) or desktop Linux. Telegram's iOS reproducible-build docs are the most complete public precedent and still describe a nontrivial manual diffing process, not an automated public verifier. | **missing**, and realistically **post-v1** given iOS tooling constraints — not a near-term blocker, but worth stating as an aspiration since it's the natural end-state of the "auditable" pitch | post-v1 (track Telegram's iOS approach as a reference if ever pursued; not worth blocking v1 on, since the underlying source-auditability + "no network code in the extension" claims are independently strong without it) |
| BÍN / Árni Magnússon Institute attribution | Already resolved in PLAN.md ("BÍN clause" — confirmed license terms navigated) and appears in SettingsView.swift's About section (`aboutBinTitle`/`aboutBinDetail`). | **covered** | done |

## Surprising findings

- **Haptics silently don't work without Full Access** — this is a real, sourced iOS platform limitation (`UIImpactFeedbackGenerator` requires Full Access inside an extension) that directly undercuts an unqualified reading of PLAN.md's "extension untouched by Full Access denial → still fully functional" claim. Needs on-device re-verification and, if confirmed, explicit user-facing messaging — otherwise a Full-Access-denied user will experience "haptics are broken" as a bug report, not an expected trade-off.
- **The emoji-keyboard non-goal is contradicted by the current code**: `KeyboardViewController.swift` still wires up KeyboardKit's stock `emojiKeyboard: { $0.view }` slot, meaning the app currently *does* ship a basic emoji picker — in tension with PLAN.md's stated non-goal ("emoji search" excluded) and its own flagged risk (GH #757 unresolved emoji font-cache memory bloat under a tight jetsam budget). This wants an explicit decision, not default inheritance from the framework.
- **No official API exists to detect "is my keyboard enabled" from the containing app** — the only known technique (`UserDefaults.standard["AppleKeyboards"]`) is undocumented and has a history of quietly breaking; this caps what the "already configured, don't nag" onboarding UX can reliably do.
- **Export/import is currently one-directional** — the app can import a SwiftKey vocabulary export but has no export of its own, which is an odd asymmetry for a project whose core differentiator is data portability/trust.
- **KeyboardKit 9.9.1 gives away more baseline table-stakes for free than PLAN.md gives it credit for**: caps lock, autocap, dark/light appearance-following, VoiceOver action labels, and landscape layout sizing all appear to already be present in the vendored framework with zero custom code — worth an explicit verification pass rather than more building, since these are "already covered, just prove it" items, not gaps.
